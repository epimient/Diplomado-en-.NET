# Dudas — Clase 20

*Espacio para anotar preguntas y notas de clase.*

## Preguntas frecuentes

### ¿Cómo estructurar la presentación final?

1. Portada (nombre del proyecto, integrantes, fecha).
2. Descripción del proyecto (2 min).
3. Tecnologías usadas y arquitectura.
4. Demostración en vivo de la API.
5. Principales desafíos y soluciones.
6. Enlace al repositorio y demo.
7. Conclusiones y aprendizajes.
8. Preguntas del público.

### ¿Qué consejos dar para la demostración en vivo?

- Ten la aplicación corriendo antes de empezar.
- Prepara datos de prueba.
- Muestra primero el caso feliz (GET, POST exitosos).
- Luego muestra validaciones (errores 400, 404).
- Ten un plan B si algo falla (capturas de pantalla).
- Habla claro y muestra confianza.

### ¿Cómo responder preguntas técnicas?

Si no sabes la respuesta, di honestamente "No lo sé, pero investigaré" en lugar de inventar. Repasa conceptos clave antes: ORM, REST, HTTP, migraciones, DTOs.

### ¿Qué criterios se evaluarán en el proyecto?

- Funcionalidad (30%): CRUD completo funcionando.
- C# y .NET (20%): código limpio, POO, buenas prácticas.
- Base de datos (15%): relaciones, migraciones, seed data.
- API y Swagger (15%): endpoints REST, documentación.
- Documentación (10%): README, capturas, video.
- Presentación (10%): claridad, organización, demostración.

### ¿Cómo grabar el video de demostración?

Usa OBS Studio (gratuito) o la grabadora de Windows/macOS. Requisitos:

- Mostrar la terminal ejecutando `dotnet run`.
- Mostrar Swagger UI probando cada endpoint.
- Mostrar el código fuente brevemente.
- Máximo 5 minutos.
- Buena iluminación y audio claro.

### ¿Qué incluir en el repositorio final?

- Código fuente completo.
- README.md profesional.
- Capturas de pantalla en `screenshots/`.
- Script de base de datos o migraciones.
- Archivo .gitignore adecuado.
- Commits frecuentes que muestren progreso.

### ¿Qué hacer si el proyecto no está completamente terminado?

Presenta lo que tengas funcionando. Explica qué falta y por qué. Destaca lo que sí lograste hacer bien. La evaluación considera el esfuerzo y aprendizaje, no solo el producto final.
